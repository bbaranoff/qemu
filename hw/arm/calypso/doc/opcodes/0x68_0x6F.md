# 0x68xx-0x6Fxx Opcode Family — Spec for QEMU C54x Decoder

**Status:** SPEC ONLY — patch not yet written.
**Source:** binutils 2.21.1 `opcodes/tic54x-opc.c` + SPRU172C
**Date:** 2026-04-28

## Why this exists

QEMU's current C54x decoder has a single fallback `(op & 0xF800) == 0x6800`
treating the entire range 0x6800-0x6FFF as `LD Smem, T` (1-word). Per binutils,
the 7 sub-families (excluding 0x6Dxx which is correctly handled) are all
**2-word instructions** with distinct semantics. The wrong fallback causes:
- PC drifts +1 word at every site (operand consumed as next opcode)
- The `lk`/`pmad` operand executes as a parasitic instruction
- 2107 sites in firmware ROM (PROM0/1/2/3) → wedge displaces between runs
  depending on which path is taken

Reference: this spec is for fixing that. See
`doc/datasheets/README.md` §7 candidate (1) "opcode mis-decode that drifts
the PC away from the init path".

## Family overview

| First-word | Mnemonic | Words | Operands (binutils) | Status QEMU |
|---|---|---|---|---|
| `0x68xx` | `ANDM #lk, Smem` | 2 | OP_lk, OP_Smem | **MISSING** |
| `0x69xx` | `ORM  #lk, Smem` | 2 | OP_lk, OP_Smem | **MISSING** |
| `0x6Axx` | `XORM #lku, Smem` | 2 | OP_lku, OP_Smem | **MISSING** |
| `0x6Bxx` | `ADDM #lk, Smem` | 2 | OP_lk, OP_Smem | **MISSING** |
| `0x6Cxx` | `BANZ pmad, Sind` | 2 | OP_pmad, OP_Sind | **MISSING** |
| `0x6Dxx` | `MAR Smem` | 1 | OP_Smem | OK |
| `0x6Exx` | `BANZD pmad, Sind` | 2 (delayed, 2 slots) | OP_pmad, OP_Sind | **MISSING** |
| `0x6Fxx` | `ADD/SUB/LD/STH/STL Smem,SHIFT,DST` | 2 (extended) | (see §6F00 sub-encoding) | **MISSING** |

Smem encoding for first word, bits 6:0:
- bit 7 = 0 → direct (DP-relative): `addr = (DP << 7) | (op & 0x7F)`
- bit 7 = 1 → indirect: standard ARn-mode (already covered by `resolve_smem`)

## §0x68xx — ANDM #lk, Smem

```
Encoding:
  word0: 0110 1000 SSSS SSSS  (op & 0xFF00 == 0x6800)
  word1: KKKK KKKK KKKK KKKK  (lk = 16-bit immediate)

Semantics:
  data[Smem] = data[Smem] & lk

PC advance: +2 (consumed=2)
Side effects: TC unchanged (per FL_NR — no result-related flag)
              No accumulator side effect.
```

binutils flags `FL_NR` = no operand requires a real-time check. Pure memory op.

## §0x69xx — ORM #lk, Smem

```
Encoding:
  word0: 0110 1001 SSSS SSSS  (op & 0xFF00 == 0x6900)
  word1: KKKK KKKK KKKK KKKK

Semantics:
  data[Smem] = data[Smem] | lk
```

`FL_NR|FL_SMR` (FL_SMR = "smul/sst result", relevant only for SXM/saturation,
not for bitwise ops — confirm before relying).

## §0x6Axx — XORM #lku, Smem

```
Encoding:
  word0: 0110 1010 SSSS SSSS  (op & 0xFF00 == 0x6A00)
  word1: KKKK KKKK KKKK KKKK  (lku = unsigned 16-bit)

Semantics:
  data[Smem] = data[Smem] ^ lku
```

binutils uses `OP_lku` (unsigned) here vs `OP_lk` (signed) in others — matters
for sign-extension semantics during the lk fetch (effectively no diff for
bitwise XOR but documented).

## §0x6Bxx — ADDM #lk, Smem

```
Encoding:
  word0: 0110 1011 SSSS SSSS  (op & 0xFF00 == 0x6B00)
  word1: KKKK KKKK KKKK KKKK

Semantics:
  data[Smem] = data[Smem] + lk

  /* TC and overflow flags affected per SXM/OVM as for arithmetic */
```

## §0x6Cxx — BANZ pmad, Sind

```
Encoding:
  word0: 0110 1100 IIII IIII  (op & 0xFF00 == 0x6C00, Sind in low bits)
  word1: PPPP PPPP PPPP PPPP  (pmad = branch target)

Semantics:
  AR_arp -= 1   (decrement current AR via Sind addressing)
  if (AR_arp != 0) PC = pmad
  else PC += 2  (fall through)
```

`B_BRANCH|FL_NR` — branch flag, no real-time gating.

**Note on Sind:** binutils `OP_Sind` = "indirect Smem only" (bit 7 = 1 required).
The decode of the AR involved follows the same indirect-mode logic as
`resolve_smem`. The decrement happens BEFORE the test (AR is post-modified or
pre-modified per the indirect mode chosen — confirm against SPRU172C).

## §0x6Exx — BANZD pmad, Sind (delayed)

Same as BANZ but with **2-slot delay**: two instructions following the BANZD
execute regardless of branch decision; branch is taken (or not) only after.

```
Encoding:
  word0: 0110 1110 IIII IIII  (op & 0xFF00 == 0x6E00)
  word1: PPPP PPPP PPPP PPPP

Semantics:
  AR_arp -= 1
  branch_taken = (AR_arp != 0)
  PC += 2  (fall to delay slot 1)
  exec delay slot 1 (1 word)
  exec delay slot 2 (1 word)
  if branch_taken: PC = pmad

Pipeline: same delay-slot semantics as BD/CALD/RETD/F6E3 CALAD.
```

`FL_DELAY` flag explicit. **Must reuse the existing delay-slot mechanism** in
QEMU (the one used for F6E3 CALAD / F073 BD / F074 CALL etc.) — DO NOT
implement a fresh path.

## §0x6Fxx — Extended ADD/SUB/LD/STH/STL Smem, SHIFT, DST/SRC

This is the tricky one. binutils marks all 5 mnemonics with the same
`base=0x6F00 mask=0xFF00` for word 0, BUT additionally specifies a
**second-word base/mask** in the `extended_opcode` and `extended_mask` fields
(the trailing `0x0Cxx, 0xFxxx, XREST` after the flags):

```
add  word0=0x6F00 mask=0xFF00   word1 base=0x0C00 mask=0xFCE0   FL_EXT|FL_SMR
sub  word0=0x6F00 mask=0xFF00   word1 base=0x0C20 mask=0xFCE0   FL_EXT|FL_SMR
ld   word0=0x6F00 mask=0xFF00   word1 base=0x0C40 mask=0xFEE0   FL_EXT|FL_SMR
sth  word0=0x6F00 mask=0xFF00   word1 base=0x0C60 mask=0xFEE0   FL_EXT
stl  word0=0x6F00 mask=0xFF00   word1 base=0x0C80 mask=0xFEE0   FL_EXT
```

Decoder logic for 0x6F00:
```
1. Smem = word0 & 0x7F  (with bit 7 = direct/indirect flag)
2. word1 = next_word
3. Match word1 against masks (most specific first):
     (word1 & 0xFCE0) == 0x0C00 → ADD Smem,SHIFT,SRC,[DST]
     (word1 & 0xFCE0) == 0x0C20 → SUB Smem,SHIFT,SRC,[DST]
     (word1 & 0xFEE0) == 0x0C40 → LD  Smem,SHIFT,DST
     (word1 & 0xFEE0) == 0x0C60 → STH SRC,SHIFT,Smem
     (word1 & 0xFEE0) == 0x0C80 → STL SRC,SHIFT,Smem
4. SHIFT  = word1 bits 4:0 (signed 5-bit, -16..+15)
5. SRC    = word1 bit 9 (0=A, 1=B)
6. DST    = word1 bit 8 (0=A, 1=B)  [for ADD/SUB; LD uses different bit?]
```

### Sub-bit assignments (need verification against SPRU172C)

For ADD/SUB (mask `0xFCE0`):
- bit 9 = SRC
- bit 8 = DST
- bits 7:5 = type-encoded zero (must be `0b110` — `0x0C00 >> 5 = 0x60`)
- bits 4:0 = SHIFT (signed 5-bit)

For LD/STH/STL (mask `0xFEE0` — different mask, bit 8 is included):
- bit 9 = SRC1/DST  (single-acc selection)
- bit 8 = part of opcode discriminator
- bits 7:5 = subop bits (010/011/100 for LD/STH/STL)
- bits 4:0 = SHIFT

**Action item:** read SPRU172C Volume 2 entries for `ADD Smem,16,SRC,DST`
(p.4-26 area) and `LD Smem,SHIFT,DST` to confirm the exact bit assignments
of word1 before implementing. The masks 0xFCE0 vs 0xFEE0 hint that bits 9-8
play different roles for ADD/SUB vs LD/STH/STL; getting this wrong will
dispatch the right mnemonic but with wrong operands.

### Wedge case verification

PROM0[0x834C] = 0x6F07
PROM0[0x834D] = 0x0C41

- word0 & 0xFF00 = 0x6F00 → 0x6Fxx family ✓
- Smem (bits 6:0) = 0x07 → direct, addr = (DP=0x01F << 7) | 0x07 = 0x0F87
- word1 & 0xFEE0 = 0x0C40 → matches **LD Smem, SHIFT, DST**
- SHIFT = word1 & 0x1F = 0x01
- DST = (word1 >> 9) & 1 = 0 → A
- Effect: `A = (data[0x0F87] << 1) sign-extended` (under SXM=1)

If A = data[0x0F87] = some real value, no self-loop wedge. ✓ Patch validated.

### MAC at 0xF067 — separate, do not include here

Note that `mac` also appears with `base=0xF067 mask=0xFCFF` — this is a
**different opcode** (in the F0xx family, not the 6F00 family). It's the
"long-immediate MAC" form `MAC #lk, SRC, [DST]`. Already in scope of the
F4-F7 generic family that was previously fixed.

## Implementation order (per user strategy)

1. **0x6F00 first** (544 hits, the highest impact and tied to the current
   wedge at PC=0x8353). Verify against the wedge case above. If `A_low` no
   longer becomes 0xFFFA → 0x8353, the diagnostic is closed.
2. **0x6800-0x6Bxx** (ANDM/ORM/XORM/ADDM, 1259 hits combined). Trivial
   2-word memory ops.
3. **0x6Cxx BANZ + 0x6Exx BANZD** (304 hits combined). MUST integrate
   with existing delay-slot mechanism.
4. **Re-test full run** after each step. Validation = hot PC distribution
   shifts away from the wedge sites.
